# Deckboss

> Your AI assistant for edge robotics and IoT systems. Clone this repo onto a Jetson or Raspberry Pi, run one command, and start talking to your hardware.

**Build your system. Deploy with Cocapn.**

## Quick Start

```bash
git clone https://github.com/Lucineer/deckboss.git
cd deckboss
bash setup.sh
deckboss
```

That's it. Deckboss boots, walks you through onboarding, and you're talking to your Jetson.

## Where Deckboss Fits

Every Lucineer app has an agent at its center — skills, context, and equipment. Most users never think about it. They use studylog.ai or activelog.ai and it just works.

Deckboss is for the people who build and maintain those systems. It's the "open the hood" layer:

```
Layer 1 — Touch     studylog.ai, activelog.ai, dmlog.ai, makerlog.ai
                    "It just works. Has an AI agent inside."

Layer 2 — Operate   cocapn.ai (runtime) / cocapn.com (membership)
                    "I can customize agents, BYOK, manage my fleet"

Layer 3 — Build     deckboss.ai (design) / deckboss.net (hardware) / capitaine.ai (premium)
                    "I design systems, equip agents, open the hood"
```

Most users live at Layer 1. Power users move to Layer 2. Developers, technicians, and system integrators work at Layer 3. Some become technicians who open the hood for others. That's the flywheel.

## First Run

Deckboss runs a 6-step onboarding wizard:

1. **Hardware Detection** — Scans your Jetson/RPi for RAM, GPU, cameras, audio, GPIO
2. **Model Engine** — Choose Ollama (serial, lightweight) or vLLM (parallel/swarm, more RAM)
3. **API Keys** — Add cloud providers (DeepSeek, z.ai, DeepInfra, SiliconFlow) or skip and use local only
4. **Input/Output** — Set up how you talk to Deckboss:
   - Terminal (default)
   - Telegram bot (talk from your phone on the boat)
   - Discord bot
   - LAN API (HTTP endpoint for local network devices)
   - Google Docs sync (monitor folder changes)
5. **Hardware I/O** — Optional: USB mic/speaker for speech-to-text and text-to-speech
6. **Profile Selection** — Choose which git-agent profile to equip

## The Character Sheet

After onboarding, Deckboss creates `~/.deckboss/character.yaml` — a resource allocation plan based on your hardware and role.

This is the key architectural concept. A Jetson with 2 CSI cameras for stereo vision in robotics arranges resources completely differently than a Cocapn acting as an ideation agent that expands repos all night. The character sheet makes these tradeoffs explicit.

```yaml
# Auto-generated by Deckboss onboarding
hardware:
  device: "Jetson Super Orin Nano"
  ram_gb: 8
  vram_gb: 6
  detected:
    csi_cameras: 0
    audio: false

resource_plan:
  model_engine: "ollama"
  max_gpu_model_size_gb: 4
  pipeline_mode: "serial"

models:
  primary:
    engine: "ollama"
    model: "phi3:mini"
    priority: 1
  reasoning:
    engine: "cloud"
    provider: "deepseek"
    model: "deepseek-reasoner"
    priority: 2

io:
  primary: "terminal"
  secondary: ["telegram"]

profile: "lucineer/marine"
role: "system-designer"
```

## Skills, Context, and Equipment

Every git-agent has three types of resources:

- **Equipment** — What the agent perceives through. Sensors, cameras, APIs, file systems. This is the pre-model input layer.
- **Skills** — How the agent thinks. Knowledge files, reasoning patterns, domain expertise. This shapes model behavior.
- **Context** — What the agent knows. Conversation history, project state, accumulated knowledge. This is the runtime working memory.

The character sheet allocates hardware resources across these three layers. More cameras means less RAM for models. More models running means less room for context. Every deployment makes different tradeoffs.

### Vibe-Coding Sessions

Deckboss uses a Claude Code-like interaction for system design:

1. Describe what you need in plain language
2. A higher-tier model (usually cloud) creates the design
3. Deckboss proposes a git-agent fleet — one repo per subsystem
4. Riff on the design — move sensors, add redundancy, change thresholds
5. Deckboss updates the character sheet and generates code
6. When ready: `deckboss handoff --to cocapn`

### Managing Agents

```bash
deckboss                    # Start the TUI
deckboss status             # Show character sheet and resource usage
deckboss agents             # List equipped agents
deckboss pull marine/engine-monitor   # Pull an agent from a profile
deckboss session engine-monitor        # Start a focused agent session
deckboss config edit         # Edit character sheet directly
deckboss profiles list       # Browse available profiles
```

## Git-Agent Architecture

Every agent is a git repo. The repo IS the agent:

```
~/.deckboss/agents/
└── engine-monitor/
    ├── README.md           # What this agent does
    ├── agent.yaml          # Config: inputs, outputs, model preferences
    ├── main.py             # Runtime
    ├── skills/             # How it thinks
    │   ├── gauge_reading.md
    │   ├── flood_detection.md
    │   └── temp_alerting.md
    └── equipment/          # What it perceives through
        └── sensors.yaml    # Connected sensor definitions
```

### Profiles

Profiles are curated collections of agents. Anyone can create one:

```bash
# Browse agents in the marine profile
deckboss profiles browse lucineer/marine

# Pull an agent
deckboss pull lucineer/marine/engine-monitor

# Add a custom profile source
deckboss profiles add acme https://github.com/acme
deckboss profiles browse acme/warehouse
```

A robotics developer creates `acme/warehouse` with agents for inventory tracking, forklift monitoring, safety compliance. A farmer creates `iowa/grain` with soil sensors, weather agents, yield prediction. Deckboss users add these profiles and equip their units.

### Hand Off to Cocapn

When your design is complete:

```bash
deckboss handoff --to cocapn
```

Packages your agent fleet for Cocapn deployment. Deckboss powers down to cold storage. Cocapn takes over.

## Example Setups

### Marine Technician (You)
Jetson Orin Nano 8GB, no cameras. Whisper STT (via phone), Phi-3-mini (local), DeepSeek-Reasoner (cloud). Terminal + Telegram. Designs marine IoT systems from the boat.

### Robotics Lab
Jetson Orin Nano 8GB, 2x CSI cameras, USB mic. Ollama for serial, vLLM for parallel camera inference. Terminal + LAN API. Real-time vision and motor control.

### Content Creator (Pure Software)
Raspberry Pi 5 8GB, no cameras. Cloud-only (DeepSeek, z.ai) — Pi handles orchestration, cloud handles thinking. Terminal + Discord.

### Night Owl
Jetson Orin Nano 8GB, no cameras. DeepSeek-Reasoner for ideation, DeepSeek-chat for bulk work. Terminal only, overnight unattended. Spawns agents, fills capability gaps, PRs all night.

```
~/.deckboss/
├── config.yaml          # Global configuration
├── character.yaml       # Hardware resource plan
├── secrets.yaml         # API keys (chmod 600)
├── profiles/            # Available agent profiles
├── agents/              # Equipped git-agent repos
├── models/              # Downloaded local models
├── logs/                # Session logs
└── sessions/            # Active vibe-coding sessions
```

## Requirements

- Python 3.10+
- Ubuntu (JetPack 6.x, Raspberry Pi OS, or any modern Linux)
- Git
- 8GB+ RAM recommended (4GB minimum)
- Optional: NVIDIA GPU (CUDA 12.x), USB mic/speaker, CSI cameras

## Install From Scratch

### Jetson (NVIDIA JetPack 6.x)
```bash
sudo apt update && sudo apt install -y python3 python3-pip git
git clone https://github.com/Lucineer/deckboss.git
cd deckboss && bash setup.sh && deckboss
```

### Raspberry Pi 5
```bash
sudo apt update && sudo apt install -y python3 python3-pip git
git clone https://github.com/Lucineer/deckboss.git
cd deckboss && bash setup.sh && deckboss
```

## Philosophy

- **The repo IS the agent.** Every capability lives in git. Fork it, improve it, share it.
- **Hardware-aware.** Character sheet adapts to what you actually have.
- **Local first.** Works without internet. Cloud is enhancement, not requirement.
- **Profile-driven.** Marine, robotics, content, farming — same architecture, different agents.
- **BYOK everywhere.** Your keys, your models, your data. Zero lock-in.
- **Under-promise, over-deliver.**
